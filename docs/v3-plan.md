# TinyHwBar 3.0.0 实施与发布边界

本文件记录从 `2.0.0` 到 `3.0.0` 的产品改动、验证边界和公开发布门禁。它不替代 GitHub Actions、Release 资产回读或逐项发布审批；正式二进制是否已经发布，以 [`v3.0.0` GitHub Release](https://github.com/Chuyu03/TinyHwBar/releases/tag/v3.0.0) 的实际可见状态为准。

## 版本重点

### GPU 默认自动选择与手动选择

- 启动后默认使用“自动选择（推荐）”，用户无需先配置 GPU。可用独显有有效数据时优先显示独显；独显处于 ECO、不可用或没有有效利用率而核显可用时，监控条自动回退核显。
- 设置页列出检测到的物理独显和核显，并提供刷新入口、当前自动使用设备和手动选择状态。手动模式保存设备 LUID 与名称；设备暂时不可用时保留偏好、临时回退自动选择，设备恢复后重试。
- GPU 枚举会在能够可靠识别时排除远程、软件渲染和虚拟显示候选，避免 ToDesk 等根枚举虚拟显示适配器借用实体 GPU 名称而制造重复候选。自动模式遇到同一角色多个可靠候选时明确显示歧义，用户可手动选择已列出的物理 GPU；仍无法可靠分类的候选不按枚举顺序猜选。

### 设置、历史与界面完整性

- 设置和历史加载使用结构化结果，区分成功、缺失、损坏、未来版本和暂时不可读。主文件异常或仅剩备份时可以只读恢复，但不会在启动或定时保存时自动覆盖异常状态；恢复会话中重新启用的网络或持久化功能只对本次运行生效，不会写回受保护的设置文件。
- 关闭历史持久化只停止后续磁盘写入，不删除旧历史；只有用户确认“清除全部本地历史”才删除主文件与备份。不完整清除会保护后续写入，避免定时保存重建半删除状态。
- 历史横轴使用采样时的真实 UTC 时间，长空档断线，乱序或时钟回拨记录不会破坏过期判断；默认 2 秒采样和 900 点上限连续运行时约覆盖 30 分钟，24 小时是单点年龄上限。
- 控制中心只对出现 ClearType 彩色边缘块的遥测复选框与状态文字启用灰度抗锯齿，并固定切换前后的首选尺寸；其他控件、字体、布局和业务行为不变。
- 控制中心首次显示时约束到当前工作区，高缩放或较小屏幕下可滚动访问完整设置；后台状态同步不会覆盖尚未点击“应用”的编辑内容。

### 运行、安装与发布安全

- 单实例范围按当前用户 SID 跨 Windows 会话；安装和卸载同时检查新版全局锁与旧版当前会话锁，只阻断当前用户或占用当前用户安装路径的进程。
- 安装器拒绝提升权限运行，并验证源 EXE 的产品身份、文件名、x64 PE/.NET 架构、版本和 SHA-256；卸载器先完成可回滚事务，只有显式请求且前序成功后才删除用户数据。
- 发布准备工具只接受人工审阅并显式传入的真实 Git for Windows 可执行文件、SHA-256 和有效签名者指纹；拒绝 PATH 自动解析和 `cmd\git.exe` shim，并递归检查 Git 安装树及其祖先目录没有向其他本机主体开放可替换工具链的权限。任何 Git 执行都必须在高影响确认之后。
- 发布准备还强制要求已存在、人工预置且通过 Owner/ACL/reparse point 复核的专用 `TrustedWorkspaceRoot`。源码仓库必须是该根的严格子目录并保留根内 `.git`；Git archive、展开源码、构建、冻结脚本的 `TEMP`/`TMP`/`TMPDIR`、ZIP 回读和 staging 全部位于同一根内。脚本不创建信任根、不修改 ACL，并在创建目录、归档、展开、构建和完成 staging 后重复 fail-closed 校验。
- 发布期间所有 Git 调用显式使用 `--no-replace-objects`、`--no-lazy-fetch`、`--no-optional-locks` 和 `--no-pager`。继承的 `GIT_*` 覆盖会先快照、清除并在结束后精确恢复；`GIT_EXEC_PATH` 固定到已批准 Git 的 `libexec\git-core`，system/global config 与 attributes 固定到信任根内空文件，每次调用绑定精确 `safe.directory` 与空 attributes。首次 Git 前和确认后都会原始复查 `.git/config` / `config.worktree`，拒绝 `include` / `includeIf`、`fsck` 覆盖、可注册自定义归档命令的 `tar.*` section、续行、NUL 与无效 UTF-8；`.git/commondir`、对象 alternates、`.git/info/grafts`、`.git/info/attributes`、`.git/shallow` 也会被直接拒绝。common directory 与 `--git-path objects` 必须分别等于根内 `.git` 和 `.git\objects`，对象库须通过 `git fsck --full --strict`，归档使用 `--no-worktree-attributes`，以保证完整提交 ID 和实际 tree 不受外部对象、配置或属性污染。
- 发布准备从指定完整提交在该受保护根的隔离子目录构建，只生成严格 allowlist 的 ZIP、全新 `SHA256SUMS.txt` 和来源记录；它不执行 `git add`、commit、push、tag、上传或 Release 发布。

## 默认通信与隐私边界

- 产品是“本地优先”而不是“完全离线”：受限的本地网关 ICMP 默认开启，每 10 秒最多向当前接口报告的本地地址发送一次请求；公网和全局地址在发包前拒绝。
- 更新与遥测端点默认为空，自动更新、回环 API 和遥测默认关闭。遥测每次发送均需重新预览和确认；程序不后台自动上传。
- 回环 API 只绑定 `127.0.0.1` 的随机高端口，并使用每次运行重新生成的令牌。其单工作线程仍可能被本机慢连接短暂占用，但不会阻塞监控采样或界面。

## 验证与正式发布门禁

最终公开提交必须重新完成以下步骤，不能复用 `2.0.0` 的证据：

1. 复核最终 diff、敏感信息、公开候选树和依赖边界；确认 `outputs/`、`release/`、本机快捷方式、临时日志和工作区路径不进入公开文件集。
2. 退出所有 TinyHwBar 实例后，运行 `test.cmd --compile-only`、完整 `test.cmd` 和 `build.cmd`；完整测试若被 Smart App Control 阻止，必须与断言失败明确区分，不能关闭系统保护。
3. 取得 `3.0.0` 对应提交的 GitHub Actions Build 成功证据。
4. 将冻结仓库置于已人工审阅的专用 `TrustedWorkspaceRoot` 严格子目录，从该完整提交在同根隔离 work/staging 子目录重新构建；核对 Owner/ACL/reparse point、ZIP allowlist、版本元数据、完整提交 ID、二进制 README 相对链接和新生成的 SHA-256。
5. 分别取得 `git add`、commit、push、tag、Draft Release、资产上传和正式发布批准；任何批准只覆盖当次展示的明确范围。
6. 发布后使用无需身份的公开页面重新下载并核对 Release、ZIP 与 `SHA256SUMS.txt`。

## 已知限制

- 每个 GPU 角色只显示一个设备；多个可靠候选或分类不确定时会显示歧义。设备 LUID 与名称是明确选择标识，但驱动重装或硬件拓扑变化后不保证永久不变。
- 顶部监控条仍为固定物理像素尺寸；Per-Monitor-V2 已启用，但不同 DPI、多显示器和完整安装生命周期仍需扩大人工覆盖。
- 当前发布路径未使用受信任发布者签名，Smart App Control 仍可能阻止未知构建。不要为运行 TinyHwBar 关闭 Smart App Control、Defender 或添加排除项。
- 可预测的跨会话 Mutex 名称可能被另一名本机用户预先创建并造成拒绝运行。这是共享或敌对多用户主机上的可用性限制，不会因此获得代码执行或读取 TinyHwBar 数据；彻底消除需要后续改用 private namespace 或等效受保护机制。
