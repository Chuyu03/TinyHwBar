本文件记录从 V1 到 `2.0.0` 的实施过程。该版本将 V1 明确不包含的功能纳入优化范围，并按“本地只读 → 本地持久化 → 用户级系统变更 → 明确确认的通信”推进。`2.0.0` 已确定为稳定版；正式二进制的可下载状态以 [GitHub Release](https://github.com/Chuyu03/TinyHwBar/releases/tag/v2.0.0) 为准，当前仍未完成受信任签名。

## 本地采样与历史

- 接收/发送速率来自连续两次本机接口累计字节计数器，是被动吞吐而不是互联网测速。
- 适配器通过 Windows 路由表分别查询固定 IPv4 / IPv6 文档保留目的地址的最佳接口；查询只读本机路由表，不发包。路由指向 VPN Tunnel 时允许选择该接口，也支持只有 IPv6 路由的环境。若两个协议族指向不同适配器，会明确降级而不猜选；结果不被表述为所有流量唯一的“当前出口”。
- 路由选择与接口累计字节计数均为本地被动读取，不发送测速流量；同时显示已确认接口的名称和其报告的标称链路速率，虚拟接口可能报告合成值。
- 所选路由接口报告的本地网关延迟检测默认开启，每 10 秒最多发送一个 ICMP Echo，等待上限 750 毫秒。若接口报告多个同地址族网关，当前版本按稳定顺序选择其一，不声称它一定是路由 next hop。仅允许 RFC1918、CGNAT `100.64.0.0/10`、IPv4 link-local、IPv6 link-local 与 ULA；公网 IPv4 或全局 IPv6 会在发包前拒绝。没有外部延迟目标。
- GPU 以角色而非厂商呈现：左侧为“独立显卡”，右侧为“核显”。程序枚举 DXGI 中的 NVIDIA、AMD、Intel 物理适配器，在能够可靠识别时排除远程、软件渲染和虚拟候选，按 LUID 去重后以 DXCore `IsIntegrated` 分类；Windows GPU Engine / Adapter Memory 计数器提供角色利用率与内存。NVIDIA NVML 仅作为 NVIDIA 独立显卡温度的可选增强。类型接口不可用、候选不唯一或指标不受支持时明确降级，不猜选设备或伪造数值。厂商和型号相同但 LUID 不同的候选不会按名称合并；只有完整设备身份一致时才视为同一适配器镜像。
- 历史默认写入 `%LOCALAPPDATA%\TinyHwBar\history.csv`，只保存数值型采样，最多 900 点、24 小时、2 MiB。文件过大或头部无效时整文件拒绝；混合文件中的非法、异常或过期记录会逐行忽略。GPU 曲线采用“独立显卡 / 核显”名称，但磁盘架构与列顺序保持兼容；角色命名变更不会主动清空已有历史，后续正常持久化仍沿用原子保存流程。只有用户主动清除时才删除历史。
- 控制中心固定为 `概览 / 历史曲线 / 设置 / 高级功能` 四页；概览与历史中的 GPU 名称统一为“独立显卡 / 核显”。
- 控制中心 GPU 卡片将核心指标与设备名称拆分：核心指标保持大字号单行且不允许省略，名称使用次要字号并可单独缩略；本机人工验收已确认独显与核显数值完整可见。
- 控制中心已显式配置键盘焦点顺序、`Enter / Esc` 行为和关键控件辅助名称；浅色界面的文字及曲线颜色按相应对比度门槛检查。

## 2.0.0 已知限制

- 每个角色只展示一个 GPU。同一角色存在多个候选或同时存在无法可靠分类的候选时，程序明确显示歧义，不按枚举顺序或型号名称自动选择。
- Manifest 已启用 Per-Monitor-V2，控制中心按 DPI 缩放；顶部监控条仍固定为 `330×36` 物理像素。不同缩放比例的多显示器组合尚未完成系统性人工覆盖。
- 回环 API 使用单一后台工作线程；启用后，本机未认证慢连接可使该可选 API 单次暂时不可用至约 5 秒，但不会阻塞监控采样或界面。
- 当前构建未签名，Smart App Control 仍可能阻止执行；正常启动若干次不能证明后续构建或其他设备一定不会被拦截。

## 高级功能硬边界

### 开机启动和安装

- 开机启动默认关闭，只有用户明确操作后才写入当前用户 `Run` 值；不同命令不会被静默覆盖或删除，UNC、映射网络盘及无法确认卷类型的程序路径会被安全拒绝。
- 安装/卸载脚本只面向当前用户，运行前应先使用 `-WhatIf`。
- 默认安装不启动应用、不启用开机启动；默认卸载保留 `%LOCALAPPDATA%\TinyHwBar` 用户数据。删除用户数据需要显式 `-RemoveUserData`。安装器和卸载器在任何实际变更前都会先持有按当前用户 SID 命名的跨会话 `Global\TinyHwBar.Maintenance.<SID>`，再持有与应用相同的当前会话 `Local\TinyHwBar.Singleton`，并在持锁后再次检查系统中正常命名的进程；前者串行化同一用户各会话中的维护脚本，后者阻止当前会话应用并发，但进程快照仍不能阻止检查后从其他会话启动的应用或识别改名副本。

### 更新

- 更新清单地址默认为空，启动后自动检查默认关闭；手动检查只在用户点击后发起。
- 更新和遥测使用同一公共 HTTPS 地址预检，拒绝 query、userinfo、fragment 和预检时已解析为非公网的目标。该预检不固定随后实际连接的地址，不能彻底消除 DNS rebinding，也不验证发布者身份；真实端点必须使用可信、稳定的公开域名。
- 发现新版本后，候选包下载仍需单独确认。下载上限 32 MiB，验证清单给出的 SHA-256 后只原子暂存。
- 当前实现不执行、安装、替换或自动回滚候选文件。清单与包的同源哈希只能证明二者一致，不能替代 Authenticode 或受信任发布者签名。

### 回环 API

- 默认关闭，只绑定 `127.0.0.1`，随机选择 `49152–65535` 高端口。
- 只接受 `GET /v1/status`，使用每次运行重新生成的 Bearer token。
- 启用状态会持久化；退出只停止本次监听，下次启动会恢复。持续关闭需取消勾选并应用。
- 响应为大小受限的数值型硬件/网络状态。`schemaVersion: 2` 使用 `discrete* / integrated*` GPU 字段；旧 `nvidia* / intel*` 数值字段只在实际厂商匹配时作为兼容别名返回，AMD 数据不会进入 NVIDIA 或 Intel 字段。`networkSelectionStatus` 固定为 `0 Available / 1 NoRoute / 2 SplitRoute / 3 RouteLookupFailed / 4 RouteInterfaceMissing / 5 CounterUnavailable / 6 Disposed`，没有自由文本适配器状态。没有 LAN 或 Internet 监听，也不是远程管理接口。

### 遥测/诊断摘要

- 默认关闭且端点为空；地址预检拒绝 query、userinfo 和 fragment。
- 数据使用固定字段和大小上限，发送前显示精确预览；当前 UI 每次尝试都会创建新预览并要求新的确认。服务层在成功发送后使预览失效，失败后的重试仍必须由调用方再次明确确认。
- 没有后台定时上传、日志目录扫描或默认运营端点。若未来加入真实运营端点，仍需单独确定运营者、保留/删除、认证、成本与隐私声明。

## 发布与签名边界

本机 Code Integrity 事件已经证明未签名临时测试 EXE 可能被 Smart App Control 阻止；禁用 SAC、Defender 或添加排除项不属于解决方案。SignPath Foundation 当前仅完成只读评估，没有申请资格、安装 GitHub App、修改 CI/secret、上传产物或执行签名。

`2.0.0` 已按上述门禁完成带醒目说明的未签名正式发布：最终公开文件集、代码/隐私/泄露审查、GitHub Actions 完整测试、最终提交重建、图标/版本元数据/安装脚本/ZIP allowlist/校验和验证，以及 push、tag、Draft Release、资产上传和正式发布的分层审批均已完成。尚未覆盖的多 DPI 或安装生命周期场景仍作为公开限制；后续版本仍须重新执行同等发布门禁。
