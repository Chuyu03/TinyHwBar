# Contributing to TinyHwBar

感谢你考虑改进 TinyHwBar。请保持变更小而清晰，并优先保护低资源占用、本地运行、失败降级和隐私边界。

## 开发环境与验证

- Windows 11 x64。
- .NET Framework 4.8 自带的 64 位 C# 编译器。
- 不需要 .NET SDK、NuGet 或第三方包。

在项目根目录运行：

```powershell
.\test.cmd --compile-only
.\test.cmd
.\build.cmd
```

构建应保持零警告、零错误并生成 `outputs\TinyHwBar.exe`。默认 `test.cmd` 会执行未签名临时 EXE；若 Smart App Control 阻止它，请记录 Code Integrity 证据并保留系统保护，不要关闭 SAC、Defender 或添加排除项，也不要把策略阻止写成测试断言失败。

正式窗口、进程、托盘、安装项和临时测试进程均使用 `TinyHwBar` 作为产品名；版本号只保留在文件元数据、变更记录和发布记录中。测试 EXE 使用 `TinyHwBar.Tests.*.exe`，不要把开发版本号重新放进可能由 Windows 安全通知展示的文件名。

安装和卸载脚本的最小安全验证应先使用 `-WhatIf`：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\Install-TinyHwBar.ps1 -WhatIf
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\Uninstall-TinyHwBar.ps1 -WhatIf
```

未经明确授权，不要执行实际安装、卸载、启动项写入或用户数据删除。

## 图标约定

项目采用极简仪表盘轮廓、单指针和少量刻度的正式图标，重点是简洁且能立即表达硬件状态，颜色不是主要约束。确定性资产位于 `assets\TinyHwBar.png` 和 `assets\TinyHwBar.ico`。如需按当前设计重新生成多尺寸 ICO：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\GenerateIcon.ps1
```

不要在普通功能 PR 中重新设计或替换图标。

## 2.x 必须保留的边界

`2.0.0` 已确立以下产品与安全边界；修改相关代码时必须保留并测试这些约束：

- 网络吞吐只读本机接口计数器；程序以固定文档保留地址分别查询 IPv4/IPv6 最佳接口，链路速率明确为所选接口的标称值，不得把结果描述成所有流量唯一的当前出口。
- 网关 ICMP 使用所选接口报告的同地址族网关并按稳定顺序选择，不保证它是路由 next hop；默认每 10 秒一次、750 毫秒超时，只允许 RFC1918、CGNAT、link-local 和 ULA，公网/全局地址在发包前拒绝。
- GPU 指标必须由 DXGI LUID 关联，并只用 DXCore 的可靠类型信息区分独显和核显；厂商或型号文本不能替代设备身份。多个候选无法唯一确定、同型号适配器的 LUID 不同或类型未知时必须明确降级，不得按名称合并或猜选。利用率取最忙物理引擎，当前内存与内存上限分开；温度不支持时明确显示，不推测。
- 历史只保存数值型采样，最多 900 点、24 小时、2 MiB，支持用户清除；不要写入进程、网址、数据包内容或用户文件。
- 控制中心 GPU 卡片必须优先完整展示利用率、显存/共享内存与可用温度；设备型号只能作为次要信息并可缩略，不得让型号挤压核心指标。
- 控制中心必须保留可预测的 `Tab / Shift+Tab` 顺序、`Enter / Esc` 操作、关键控件辅助名称，以及文字 4.5:1、非文本曲线 3:1 的最低对比度。
- 修改 DPI、窗口位置或布局后，应覆盖 `100% / 125% / 150% / 175% / 200%` 以及不同缩放比例的多显示器；未实际验证的组合必须在 PR 中注明。
- 开机启动、自动更新检查、回环 API 和遥测默认关闭。
- 更新或遥测 HTTPS 地址不得包含 query、userinfo 或 fragment，默认端点必须留空。公共 HTTPS DNS 预检不固定实际连接地址，不能彻底消除 DNS rebinding；真实端点只能使用可信、稳定的公开域名。不要在仓库中加入默认私有端点、凭据或 token。
- 更新包每次下载前确认、最大 32 MiB、验证 SHA-256，只暂存；不得自动执行、安装、替换或把清单哈希描述成发布者签名。
- API 只能绑定 `127.0.0.1` 随机高端口，只提供 `/v1/status`，每次运行生成新 token，只返回有界数值。启用状态持久化后，下次启动会恢复监听；永久关闭必须取消勾选并应用。
- 遥测/诊断摘要必须先展示精确字段预览，每次发送单独确认；不得后台上传。
- 安装器仅面向当前用户；默认不启动，也不修改既有启动项状态。卸载默认保留用户数据；递归删除整个用户数据目录必须显式选择并明确该删除无法回滚。安装器和卸载器在任何实际变更前都必须先持有按当前用户 SID 命名的跨会话 `Global\TinyHwBar.Maintenance.<SID>`，再持有与应用相同的当前会话 `Local\TinyHwBar.Singleton`，并在持锁后再次检查系统中正常命名的进程；跨会话 Mutex 必须串行化同一用户的维护脚本，但进程快照仍不能阻止检查后从其他会话启动的应用或识别改名副本。
- 对外介绍应使用“本地优先”，不得写成“完全离线”：本地网关 ICMP 默认开启；更新与遥测端点为空，其他外部通信功能默认关闭。

涉及网络、注册表、安装、卸载或外部服务的测试，应优先使用注入的假实现，不得在普通测试中创建真实外部请求或写入真实用户设置。

## 提交变更

1. Fork 仓库并从 `main` 创建短小、单一目的的分支。
2. 不要提交 `outputs/`、本地设置、历史、日志、凭据、暂存更新或个人路径。
3. 对硬件采样、P/Invoke、持久化、网络、构建脚本或工作流的修改必须说明安全影响和失败回退。
4. 在 PR 中列出实际运行的构建、测试和手动验证；不要把“可以编译”描述成“代码安全”或“Smart App Control 已解决”。
5. SignPath Foundation 目前仅为评估项。未经单独审批，不要申请、安装 App、添加 secret、上传产物或签名。

仓库的 GitHub Actions 设置应保持“外部贡献者工作流先由维护者审查并批准”的门槛，并由维护者定期在仓库设置中核验；该门槛不由当前工作流 YAML 单独保证。任何工作流、签名或发布脚本改动都视为高风险范围。

## 报告安全问题

不要在公开 Issue 中披露可利用细节、凭据或敏感日志。请按 [SECURITY.md](SECURITY.md) 使用 GitHub 私密漏洞报告。
